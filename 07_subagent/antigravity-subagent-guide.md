# Antigravity 서브에이전트 가이드

## 학습 목표

- Antigravity의 Agent Manager 기반 멀티에이전트 아키텍처를 이해한다
- Mission Control 대시보드에서 여러 에이전트를 시각적으로 관리하는 방법을 익힌다
- 비동기 병렬 에이전트 스폰을 통한 아이디어 팩토리와 멀티 버그 수정 패턴을 실습한다
- 아티팩트(계획서, 결과물, diff) 관리와 승인 워크플로우를 습득한다
- 메모리 및 리소스 관리를 통해 안정적인 운영 방법을 파악한다

---

## 전체 흐름 한눈에 보기

```
Antigravity Agent Manager 아키텍처
─────────────────────────────────────────────────────────────────
[ Mission Control 대시보드 ]
        │
        ├── Agent Manager
        │       │
        │       ├── 에이전트 1 (독립 컨텍스트) ─ 랜딩 페이지
        │       ├── 에이전트 2 (독립 컨텍스트) ─ 백엔드 API
        │       ├── 에이전트 3 (독립 컨텍스트) ─ 로고 & 브랜딩
        │       └── 에이전트 N (독립 컨텍스트) ...
        │
        ├── 아티팩트 관리
        │       ├── 계획서 (Plans)
        │       ├── 결과물 (Results)
        │       └── 코드 변경 (Diffs)
        │
        └── 승인 관리
                ├── 보류 중인 승인 요청
                └── 일괄 승인 / 거부
```

## Phase 1: 개념 이해

### Antigravity의 서브에이전트 지원 수준

| 항목 | 내용 |
|------|------|
| 네이티브 서브에이전트 | Agent Manager 기반 완전 지원 |
| 병렬 실행 수 | 이론적으로 제한 없음 (메모리 제약) |
| 자동 태스크 분배 | 있음 (사용자가 명시적으로 에이전트 생성) |
| UI | 시각적 Mission Control 대시보드 |
| 컨텍스트 격리 | 각 에이전트 완전 독립 컨텍스트 |
| 아티팩트 관리 | 계획서/결과물/diff 자동 생성 및 관리 |
| 승인 워크플로우 | 보류 중인 요청 시각적 관리 |
| 비용 | 유휴 상태 1GB+ 메모리 사용 |

Antigravity는 6가지 도구 중 가장 직관적인 멀티에이전트 UI를 제공한다. CLI 명령어를 외울 필요 없이, 대시보드에서 버튼 클릭만으로 에이전트를 스폰하고 관리할 수 있다. Google이 개발한 에디터 네이티브 도구로, VS Code 기반 환경에서 동작한다.

### Agent Manager의 핵심 개념

#### Mission Control 대시보드

Antigravity의 중앙 제어 인터페이스다. 다음 정보를 실시간으로 표시한다:

- **실행 중인 에이전트 목록**: 각 에이전트의 현재 상태 (실행 중/완료/오류/대기)
- **진행률 표시**: 에이전트별 작업 진행 상황
- **보류 중인 승인**: 에이전트가 파일 수정 또는 명령 실행 전 사용자 승인을 요청한 목록
- **아티팩트 보기**: 각 에이전트가 생성한 계획서, 결과물, diff

#### 에이전트 독립 컨텍스트

각 에이전트는 완전히 독립된 컨텍스트를 가진다:

```
에이전트 A의 컨텍스트:        에이전트 B의 컨텍스트:
  - src/auth/ 파일들            - src/payment/ 파일들
  - auth 관련 테스트            - payment 관련 테스트
  - JWT 라이브러리 문서          - Stripe API 문서
  (에이전트 B와 완전 격리)       (에이전트 A와 완전 격리)
```

#### 아티팩트 시스템

에이전트가 작업 중 자동으로 생성하는 산출물들이다:

- **Plans (계획서)**: 에이전트가 작업 시작 전 생성하는 단계별 실행 계획
- **Results (결과물)**: 작업 완료 후 생성되는 요약 보고서
- **Diffs (코드 변경)**: 파일 변경 사항의 diff 뷰

## Phase 2: 서브에이전트 설정 및 실행

### Agent Manager 열기

```
Antigravity 사이드바 → Agent Manager 아이콘 클릭
또는
Cmd/Ctrl + Shift + P → "Antigravity: Open Agent Manager"
```

### 에이전트 스폰 방법

**방법 1: Mission Control에서 직접 생성**

```
Agent Manager → [+ New Agent] 버튼 클릭
  → 에이전트 이름 입력 (예: "auth-refactor")
  → 초기 작업 지시 입력
  → 작업 범위(파일/디렉토리) 지정
  → [Start] 클릭
```

**방법 2: 대화 중 에이전트 분기**

```
현재 대화 창에서:
> 이 작업을 별도 에이전트로 분리해서 처리해줘

[Spawn as New Agent] 버튼 클릭 → 새 에이전트 시작
```

**방법 3: 명령 팔레트 사용**

```
Cmd/Ctrl + Shift + P → "Antigravity: Spawn Agent"
  → 작업 설명 입력
  → 에이전트 자동 생성
```

### 에이전트에게 작업 할당

각 에이전트에게 명확한 책임 범위를 지정한다.

```
에이전트 1: "auth-refactor"
  지시: "src/auth/ 디렉토리의 JWT 검증 로직을 리팩토링해줘.
         보안 취약점도 함께 점검하고 수정 계획을 먼저 보여줘."
  범위: src/auth/, tests/auth/

에이전트 2: "payment-test"
  지시: "src/payment/ 모듈의 통합 테스트를 작성해줘.
         Stripe 웹훅 처리 시나리오를 포함해줘."
  범위: src/payment/, tests/payment/

에이전트 3: "docs-update"
  지시: "API 변경사항을 반영해서 README와 API.md를 업데이트해줘."
  범위: README.md, docs/
```

### 승인 워크플로우 설정

에이전트의 자율성 수준을 설정한다.

```
Agent Manager → 에이전트 설정 → Approval Mode

옵션:
  - Ask Always: 모든 파일 수정/명령 실행 전 승인 요청
  - Ask for Risky: 파일 삭제, 외부 명령 실행 시만 승인 요청
  - Auto Approve: 모든 작업 자동 승인 (신뢰할 수 있는 환경에서만)
```

### 체크포인트 2

- [ ] Agent Manager에서 에이전트를 생성하고 작업을 할당해봤다
- [ ] 각 에이전트의 작업 범위(파일/디렉토리)를 명확히 지정했다
- [ ] 승인 모드를 적절하게 설정했다

## Phase 3: 병렬 처리 패턴

### 패턴 1: 아이디어 팩토리 (창의적 병렬 탐색)

여러 에이전트가 동일한 목표에 대해 서로 다른 접근 방식을 독립적으로 탐색한다.

```
목표: 신규 서비스 MVP 기획

에이전트 1 "landing-design":
  지시: "랜딩 페이지 디자인 시안 3가지를 HTML/CSS 프로토타입으로 만들어줘.
         각각 모던/미니멀/볼드 스타일로 구분해줘."
  → 아티팩트: landing-v1.html, landing-v2.html, landing-v3.html

에이전트 2 "api-structure":
  지시: "REST API 구조를 설계해줘. 인증, 사용자 관리, 핵심 비즈니스 로직을
         포함한 OpenAPI 3.0 스펙 초안을 작성해줘."
  → 아티팩트: openapi.yaml

에이전트 3 "branding":
  지시: "로고 SVG 3가지 변형과 컬러 팔레트를 생성해줘.
         각각 Tech/Friendly/Premium 포지셔닝에 맞게 제안해줘."
  → 아티팩트: logo-tech.svg, logo-friendly.svg, logo-premium.svg

결과: 3명의 에이전트가 동시에 서로 다른 관점의 MVP 아이디어를 제시
사용자: Mission Control에서 3개 결과를 비교하며 최적 조합 선택
```

### 패턴 2: 멀티 버그 동시 수정

독립적인 버그들을 에이전트가 동시에 처리한다.

```
GitHub Issues에서 버그 목록:
  #123: 로그인 버튼 클릭 후 500 에러
  #124: 결제 완료 후 이메일 미발송
  #125: 대용량 파일 업로드 시 타임아웃
  #126: 모바일 레이아웃 깨짐

에이전트 1 "fix-login":
  지시: "이슈 #123 수정. 로그인 엔드포인트 에러 로그를 먼저 확인하고
         근본 원인을 찾아 수정해줘. 재현 테스트도 추가해줘."

에이전트 2 "fix-email":
  지시: "이슈 #124 수정. 결제 완료 이벤트 핸들러와 이메일 서비스
         연동 코드를 확인해줘. 단위 테스트 먼저 작성 후 수정해줘."

에이전트 3 "fix-upload":
  지시: "이슈 #125 수정. 파일 업로드 스트리밍 처리로 변경하고
         타임아웃 설정을 검토해줘."

에이전트 4 "fix-mobile":
  지시: "이슈 #126 수정. 반응형 CSS를 점검하고 모바일 뷰포트에서
         깨지는 컴포넌트를 수정해줘."

4개 에이전트 동시 실행 → 예상 시간 75% 단축
```

### 패턴 3: 병렬 코드 리뷰

```
에이전트 1 "review-security":
  지시: "PR #456의 변경 파일들 중 보안 관련 코드만 집중 리뷰해줘.
         인증, 인가, 입력 검증, SQL 인젝션 관점에서 검토해줘."

에이전트 2 "review-performance":
  지시: "PR #456의 변경 파일들을 성능 관점에서 리뷰해줘.
         N+1 쿼리, 불필요한 루프, 메모리 누수를 찾아줘."

에이전트 3 "review-tests":
  지시: "PR #456의 테스트 코드만 리뷰해줘. 엣지 케이스 누락,
         모킹 오용, 불필요한 테스트를 지적해줘."

→ Mission Control에서 3가지 관점 통합 → 종합 리뷰 생성
```

### 패턴 4: 마이그레이션 분할 처리

```
에이전트 1 "migrate-core":
  지시: "src/core/ 디렉토리의 JS를 TypeScript로 변환해줘.
         strict 모드 적용, 타입 정의 포함."

에이전트 2 "migrate-utils":
  지시: "src/utils/ 디렉토리의 JS를 TypeScript로 변환해줘.
         유틸 함수 타입을 제네릭으로 설계해줘."

에이전트 3 "migrate-hooks":
  지시: "src/hooks/ 디렉토리의 JS를 TypeScript로 변환해줘.
         React 훅 타입 정의를 포함해줘."

→ 세 에이전트 동시 실행 → 각 완료 후 Mission Control에서 diff 검토 → 순차 merge
```

### 체크포인트 3

- [ ] 여러 독립적인 버그를 에이전트에 분배하여 동시 수정했다

## Phase 4: 실전 시나리오

### 시나리오 1: 스프린트 시작 일괄 작업

```
스프린트 계획:
  - 기능 A 구현 (백엔드)
  - 기능 A 구현 (프론트엔드)
  - 이전 스프린트 기술 부채 해소
  - 테스트 커버리지 80% 달성

에이전트 배치:
  에이전트 1 "feature-a-backend":
    지시: "PRD를 읽고 기능 A의 API 엔드포인트를 TDD로 구현해줘."
    범위: src/api/, tests/api/

  에이전트 2 "feature-a-frontend":
    지시: "Figma 디자인을 읽고 기능 A의 React 컴포넌트를 구현해줘.
           TypeScript + Storybook 스토리 포함."
    범위: src/components/, stories/

  에이전트 3 "tech-debt":
    지시: "TODO 주석과 FIXME 주석이 달린 코드를 모두 수정해줘.
           수정 전 각각의 영향 범위를 먼저 확인해줘."
    범위: 전체 src/

  에이전트 4 "test-coverage":
    지시: "커버리지 80% 미달 파일에 테스트를 추가해줘.
           jest --coverage 결과를 기준으로 우선순위를 정해줘."
    범위: tests/

Mission Control에서 4개 에이전트 동시 모니터링
```

### 시나리오 2: 긴급 프로덕션 대응

```
장애 상황: API 서버 응답 시간 급증 (정상 200ms → 현재 5000ms)

즉시 에이전트 3개 동시 스폰:

에이전트 1 "diagnose-db":
  지시: "슬로우 쿼리 로그와 explain analyze 결과를 분석해줘.
         응답 시간이 급증한 쿼리를 찾고 즉시 최적화 방안 제시."

에이전트 2 "diagnose-memory":
  지시: "메모리 사용량 추이와 GC 로그를 분석해줘.
         메모리 누수나 캐시 문제가 있는지 확인해줘."

에이전트 3 "diagnose-logs":
  지시: "최근 1시간 애플리케이션 로그에서 에러와 경고를
         집계해줘. 반복되는 패턴을 찾아줘."

→ 3개 에이전트 결과를 Mission Control에서 동시 확인
→ 근본 원인 파악 속도 3배 향상
→ 핫픽스 에이전트 추가 스폰
```

## 리소스 관리

### 메모리 사용량 이해

```
Antigravity 에이전트 메모리 사용:
  - 유휴 에이전트 1개: ~1GB RAM
  - 활성 에이전트 1개: ~2~4GB RAM
  - 5개 에이전트 동시: ~10~20GB RAM

권장 시스템 사양:
  - 2~3개 동시: 16GB RAM 권장
  - 4~6개 동시: 32GB RAM 권장
  - 7개 이상: 64GB RAM 이상
```

### 에이전트 수 제한 가이드

```
시스템 RAM 16GB:  최대 3개 에이전트 동시 실행
시스템 RAM 32GB:  최대 5~6개 에이전트 동시 실행
시스템 RAM 64GB+: 최대 8~10개 에이전트 동시 실행
```

### 완료된 에이전트 정리

```

→ 아티팩트는 보존, 메모리에서 해제
```

## 주의사항 및 모범 사례

### 메모리 관리

```
주의: 에이전트를 무제한으로 스폰하면 시스템이 불안정해진다.

좋은 습관:
  1. 작업 완료 후 에이전트를 Archive한다
  2. 동시 에이전트 수를 시스템 RAM에 맞게 제한한다
  3. 장기 유휴 에이전트는 종료하고 필요할 때 재시작한다
```

### 작업 범위 격리

```
나쁜 예:
  에이전트 1: "전체 코드베이스를 개선해줘" (범위 불명확)
  에이전트 2: "버그 수정해줘" (범위 불명확)
  → 두 에이전트가 같은 파일을 수정해 충돌 발생

좋은 예:
  에이전트 1: "src/auth/에서만 작업, 다른 파일 수정 금지"
  에이전트 2: "src/payment/에서만 작업, 다른 파일 수정 금지"
  → 독립적 작업, 충돌 없음
```

### 보안 고려사항

- `Auto Approve` 모드는 반드시 안전한 로컬 환경에서만 사용한다
- 에이전트가 외부 명령을 실행할 때(npm install, git push 등) 내용을 확인한다
- 민감한 정보가 있는 파일(`.env`, `credentials/`)은 에이전트 작업 범위에서 제외한다

### 비용 최적화

- 여러 에이전트가 동일한 코드베이스를 각각 인덱싱하면 비용이 중복된다
- 초기 코드베이스 분석은 하나의 에이전트가 수행하고 결과를 공유한다
- 에이전트 간 독립성이 높을수록 병렬 처리의 효과가 크다

### 모범 사례 요약

1. **범위 먼저**: 각 에이전트 생성 전 담당 파일/디렉토리를 명시한다
2. **아티팩트 활용**: Plans를 검토한 후 실행 승인한다
3. **메모리 모니터링**: 시스템 모니터에서 RAM 사용량을 확인한다
4. **독립성 극대화**: 에이전트 간 공유 파일은 최소화한다
5. **승인 모드 선택**: 실험적 작업은 Ask Always, 반복 작업은 Auto Approve
