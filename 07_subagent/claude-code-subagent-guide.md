# Claude Code 서브에이전트 가이드

## 학습 목표

- Claude Code의 Task() 기반 서브에이전트 아키텍처를 이해한다
- Explore, Plan, General 등 내장 서브에이전트 유형의 특성과 적합한 사용처를 파악한다
- 최대 7개 병렬 서브에이전트를 효과적으로 활용하는 패턴을 익힌다
- Agent Teams 기능으로 여러 Claude 세션을 협업시키는 방법을 습득한다
- 모델 라우팅 전략으로 비용과 품질을 최적화한다

---

## 전체 흐름 한눈에 보기

```
Claude Code 서브에이전트 아키텍처
─────────────────────────────────────────────────────────────────
사용자 요청
    │
    ▼
[ 메인 세션 (Opus/Sonnet) ]
    │
    ├── Task() 호출
    │       │
    │       ├── 서브에이전트 1 (Explore, Haiku) ─ 파일 탐색
    │       ├── 서브에이전트 2 (Plan, Sonnet)  ─ 계획 수립
    │       ├── 서브에이전트 3 (General, Opus) ─ 구현
    │       └── 서브에이전트 N ... (최대 7개 병렬)
    │
    │       결과 요약만 메인 컨텍스트로 반환
    │       (원시 탐색 데이터는 버림 → 컨텍스트 보호)
    │
    ▼
[ 통합 결과 사용자에게 전달 ]
```

## Phase 1: 개념 이해

### Claude Code의 서브에이전트 지원 수준

| 항목 | 내용 |
|------|------|
| 네이티브 서브에이전트 | Task() API로 내장 지원 |
| 병렬 실행 수 | 최대 7개 동시 실행 |
| 자동 태스크 분배 | 있음 (메인 에이전트가 자동 분배) |
| 결과 자동 취합 | 있음 (요약만 메인 컨텍스트로 반환) |
| 컨텍스트 격리 | 각 서브에이전트 독립 컨텍스트 |
| 메인 세션 의존성 | 메인 세션 중단 후에도 계속 실행 가능 |
| 모델 선택 | 서브에이전트별로 다른 모델 지정 가능 |

Claude Code는 CLI 기반 도구 중 가장 강력한 서브에이전트 시스템을 갖추고 있다. Task() 기반으로 여러 독립 에이전트를 스폰하고, 각 에이전트가 자신의 컨텍스트에서 작업을 완료한 후 요약만을 메인 세션으로 반환한다.

### 내장 서브에이전트 유형

#### Explore (탐색 에이전트)

파일 시스템 탐색과 코드베이스 구조 파악에 특화되어 있다. 빠른 처리가 목표이므로 경량 모델(Haiku)이 권장된다.

주요 작업:
- 디렉토리 구조 파악
- 관련 파일 수집
- 심볼/함수 위치 파악
- 의존성 트리 구성

#### Plan (계획 에이전트)

코드베이스 리서치를 바탕으로 구현 계획을 수립한다. 중간 모델(Sonnet)이 적합하다.

주요 작업:
- 영향 범위 분석
- 리팩토링 순서 결정
- 테스트 전략 수립
- 위험 요소 식별

#### General (범용 에이전트)

멀티스텝의 실제 구현 작업에 사용된다. 복잡한 작업에는 강력한 모델(Opus)이 적합하다.

주요 작업:
- 코드 작성 및 수정
- 테스트 코드 생성
- 리팩토링 실행
- 문서 작성

### Task() 메커니즘의 핵심 원리

```
메인 세션 컨텍스트 (100k 토큰 예산)
    │
    │ Task() 호출 → 서브에이전트 생성
    │
    ├── 서브에이전트 A (독립 컨텍스트, 100k 토큰)
    │       src/ 디렉토리 전체 탐색 (50k 토큰 소비)
    │       → 요약 500토큰만 반환
    │
    ├── 서브에이전트 B (독립 컨텍스트, 100k 토큰)
    │       테스트 파일 모두 읽기 (70k 토큰 소비)
    │       → 요약 300토큰만 반환
    │
    메인 세션 컨텍스트 증가: +800토큰만 (50k + 70k 대신)
```

이 구조가 왜 중요한가: 대형 코드베이스를 다룰 때 메인 컨텍스트가 빠르게 소진되는 문제를 방지한다.

### 체크포인트 1

- [ ] Task() 기반 서브에이전트의 컨텍스트 격리 원리를 이해했다
- [ ] Explore, Plan, General 각 에이전트의 적합한 사용처를 파악했다
- [ ] 결과 요약만 반환하는 방식이 컨텍스트 보호에 어떻게 기여하는지 설명할 수 있다

## Phase 2: 서브에이전트 설정 및 실행

### 기본 사용법: 자동 서브에이전트 활성화

Claude Code는 복잡한 요청을 감지하면 자동으로 서브에이전트를 스폰한다.

```bash
# Claude Code 실행
claude

# 복잡한 요청 입력 → 자동으로 서브에이전트 활성화
> auth 모듈 전체를 리팩토링하고, 단위 테스트도 함께 작성해줘.
  JWT 검증 로직 보안 취약점도 점검해줘.
```

Claude Code는 이 요청을 분석하고 내부적으로 다음과 같이 분배한다:

```
서브에이전트 1 (Explore, Haiku): auth/ 디렉토리 구조 탐색
서브에이전트 2 (Explore, Haiku): 기존 테스트 파일 탐색
서브에이전트 3 (Plan, Sonnet): 리팩토링 + 테스트 전략 수립
서브에이전트 4 (General, Opus): JWT 보안 취약점 점검
서브에이전트 5 (General, Opus): 리팩토링 실행
서브에이전트 6 (General, Sonnet): 단위 테스트 작성
```

### CLAUDE.md로 서브에이전트 동작 커스터마이징

프로젝트 루트의 `CLAUDE.md`에 서브에이전트 관련 지침을 추가할 수 있다.

```markdown
# CLAUDE.md

## 서브에이전트 사용 지침

### 탐색 작업
- src/ 탐색 시: Explore 에이전트 사용, Haiku 모델
- 테스트 탐색 시: Explore 에이전트, 최대 2개 병렬

### 계획 수립
- 영향 범위 3개 이상인 경우: Plan 에이전트 필수
- 보안 관련 변경 시: Plan + General 순차 실행

### 병렬 실행 제한
- 최대 동시 서브에이전트: 5개 (비용 관리)
- 각 서브에이전트 최대 컨텍스트: 50k 토큰
```

### 명시적 서브에이전트 요청

사용자가 직접 서브에이전트 활용을 유도할 수 있다.

```bash
# 병렬 처리를 명시적으로 요청
> 다음 세 작업을 병렬로 처리해줘:
  1. src/auth/ 리팩토링
  2. src/api/ 에러 핸들링 개선
  3. 전체 테스트 커버리지 보고서 생성

# 모델 지정 요청
> Explore 에이전트로 전체 코드베이스 구조를 파악하고,
  Haiku를 사용해서 빠르게 처리해줘
```

## Phase 3: 병렬 처리 패턴

### 패턴 1: 탐색 → 계획 → 실행 (표준 패턴)

```
요청: "이 코드베이스의 성능 병목을 찾아 개선해줘"

Phase 1 (병렬 탐색):
  서브에이전트 A (Explore): API 엔드포인트 목록 수집
  서브에이전트 B (Explore): 데이터베이스 쿼리 패턴 수집
  서브에이전트 C (Explore): 캐싱 레이어 구조 파악
  ↓ (각각 요약만 반환)

Phase 2 (계획 수립):
  서브에이전트 D (Plan): 세 탐색 결과를 통합해 최적화 순서 결정
  ↓

Phase 3 (병렬 구현):
  서브에이전트 E (General): DB 쿼리 최적화
  서브에이전트 F (General): API 응답 캐싱 추가
  ↓

메인 세션: 결과 통합 → 사용자에게 보고
```

### 패턴 2: 독립 모듈 병렬 처리

서로 의존성이 없는 모듈들을 동시에 처리한다.

```
요청: "auth, payment, notification 세 모듈에 각각 통합 테스트 작성해줘"

병렬 실행 (동시):
  서브에이전트 1: auth 모듈 탐색 + 통합 테스트 작성
  서브에이전트 2: payment 모듈 탐색 + 통합 테스트 작성
  서브에이전트 3: notification 모듈 탐색 + 통합 테스트 작성
  ↓ (세 결과 동시 반환)

메인 세션: PR 생성 및 요약 보고
```

### 패턴 3: 분산 코드 리뷰

대규모 PR을 병렬로 리뷰한다.

```
요청: "이번 PR의 변경된 15개 파일을 전부 코드 리뷰해줘"

서브에이전트 1 (Haiku): 파일 1~5 리뷰 (스타일, 간단한 버그)
서브에이전트 2 (Haiku): 파일 6~10 리뷰
서브에이전트 3 (Haiku): 파일 11~15 리뷰
서브에이전트 4 (Sonnet): 전체 아키텍처 관점 리뷰
서브에이전트 5 (Sonnet): 보안 관점 리뷰
  ↓

메인 세션 (Opus): 모든 리뷰를 통합해 최종 리뷰 코멘트 생성
```

### 패턴 4: 오류 진단 분산 수집

복잡한 버그를 여러 각도에서 동시에 조사한다.

```
요청: "프로덕션에서 간헐적으로 발생하는 500 에러를 분석해줘"

서브에이전트 1 (Explore): 에러 로그 패턴 분석
서브에이전트 2 (Explore): 최근 배포 변경사항 탐색
서브에이전트 3 (Explore): DB 슬로우 쿼리 로그 분석
서브에이전트 4 (Explore): 외부 API 연동 타임아웃 패턴 분석
  ↓ (병렬 탐색, 각 요약 반환)

서브에이전트 5 (Plan): 근본 원인 가설 수립 및 검증 계획
  ↓

메인 세션: 최종 진단 보고서 + 핫픽스 방향 제시
```

### 체크포인트 3

- [ ] 탐색→계획→실행 표준 패턴을 실제 작업에 적용해봤다
- [ ] 독립 모듈 병렬 처리 패턴으로 작업 시간을 단축해봤다
- [ ] 5개 이상의 서브에이전트를 조율하는 복잡한 워크플로우를 설계했다

## Phase 4: 실전 시나리오

### 시나리오 1: 대형 리팩토링

```
작업: "Express.js 백엔드를 Fastify로 마이그레이션해줘"

서브에이전트 1 (Explore, Haiku): 현재 Express 라우터 목록 수집
서브에이전트 2 (Explore, Haiku): 미들웨어 목록 및 의존성 수집
서브에이전트 3 (Explore, Haiku): 현재 테스트 목록 수집
  ↓ (병렬, 빠름)

서브에이전트 4 (Plan, Sonnet):
  - Express → Fastify 마이그레이션 매핑 테이블 생성
  - 우선순위 순서 결정 (코어 라우터 → 미들웨어 → 플러그인)
  ↓

서브에이전트 5 (General, Opus): 코어 라우터 마이그레이션
서브에이전트 6 (General, Sonnet): 미들웨어 변환
서브에이전트 7 (General, Sonnet): 기존 테스트 업데이트
  ↓ (병렬 실행, 7개 동시)

메인 세션: 마이그레이션 완료 확인 + 실행 테스트
```

### 시나리오 3: Agent Teams 활용 (2026.02~)

Agent Teams는 여러 독립 Claude 세션이 메시지를 주고받으며 작업을 분담하는 기능이다.

**상사-부하 관계 패턴:**

```bash
# 리드 에이전트 세션
claude --role lead
> 전체 MVP 기능 구현을 지휘해줘.
  backend-agent에게 API 구현을 지시하고,
  frontend-agent에게 컴포넌트 구현을 지시해.
  완료 보고를 받으면 통합 테스트를 진행해.

# 백엔드 에이전트 세션 (별도 터미널)
claude --role backend --team-member
> 리드 에이전트의 지시를 받아 API를 구현할 준비가 됐어.

# 프론트엔드 에이전트 세션 (별도 터미널)
claude --role frontend --team-member
> 리드 에이전트의 지시를 받아 UI 컴포넌트를 구현할 준비가 됐어.
```

**동료 협업 패턴:**

```bash
# 에이전트 A: 프론트엔드 전문
claude --team-id project-x --member-id frontend
> API 인터페이스가 결정되면 알려줘. 그에 맞춰 axios 클라이언트 작성할게.

# 에이전트 B: 백엔드 전문
claude --team-id project-x --member-id backend
> API 스펙 확정했어. POST /api/users 엔드포인트 추가했고,
  응답 형태는 { id, email, createdAt }야. frontend에게 전달해줘.
```

## 모델 라우팅 전략

비용과 성능의 균형을 맞추는 핵심 전략이다.

| 작업 유형 | 권장 모델 | 이유 |
|-----------|-----------|------|
| 파일 탐색, 구조 파악 | Claude Haiku | 빠르고 저렴, 단순 탐색에 충분 |
| 코드 분류, 패턴 매칭 | Claude Haiku | 고속 처리, 정확도 충분 |
| 계획 수립, 코드 리뷰 | Claude Sonnet | 중간 복잡도, 균형 잡힌 성능 |
| 리팩토링, 아키텍처 설계 | Claude Sonnet | 품질과 비용의 균형 |
| 핵심 구현, 보안 분석 | Claude Opus | 최고 품질 필요 |
| 복잡한 알고리즘 설계 | Claude Opus | 깊은 추론 필요 |

**비용 최적화 예시:**

```
최적화 전: 모든 서브에이전트에 Opus 사용
  7개 × Opus = 높은 비용

최적화 후:
  탐색 3개 × Haiku = 낮은 비용
  계획 1개 × Sonnet = 중간 비용
  구현 3개 × Opus = 높은 비용 (핵심만)
  → 전체 비용 40~60% 절감
```

## 주의사항 및 모범 사례

### 비용 관리

- 서브에이전트 수를 무제한으로 늘리지 않는다. 각 서브에이전트는 독립 API 호출을 발생시킨다
- Explore 에이전트는 항상 Haiku를 사용한다. 파일 탐색에 Opus는 과도하다
- CLAUDE.md에 `최대 동시 서브에이전트: N` 제한을 명시한다

### 컨텍스트 설계

```
나쁜 예: 메인 에이전트에 모든 파일을 직접 읽게 하기
→ 메인 컨텍스트 빠르게 소진

좋은 예: Explore 서브에이전트에 탐색을 위임하고 요약만 받기
→ 메인 컨텍스트 절약, 더 많은 작업 가능
```

### 서브에이전트 간 의존성 관리

- 병렬 서브에이전트들이 동일한 파일을 수정하면 충돌이 발생한다
- 각 서브에이전트의 작업 범위(담당 파일, 담당 모듈)를 명확히 분리한다
- 공유 파일이 있다면 순차 실행을 선택한다

```bash
# 충돌 방지: 담당 범위 명시
> 다음 세 작업을 병렬로 처리해줘. 각 에이전트는 해당 범위만 수정해:
  1. src/auth/ 만 수정 (JWT 리팩토링)
  2. src/api/ 만 수정 (에러 핸들링)
  3. tests/ 만 수정 (테스트 추가)
  (공유 utils/ 파일은 수정하지 말 것)
```

### 메인 세션 중단 후 재연결

```bash
# 실행 중인 서브에이전트 상태 확인
claude --status

# 중단된 세션 재연결
claude --resume [session-id]
```

### 모범 사례 요약

1. **탐색은 Haiku로**: 비용 절감의 핵심
2. **범위를 명확히**: 서브에이전트별 담당 파일/모듈을 지정
3. **7개 이하로**: 동시 7개 초과 시 순차 배치 처리
4. **결과 요약 활용**: 서브에이전트에 "결과를 500토큰 이내 요약으로 반환"을 명시
5. **CLAUDE.md 설정**: 프로젝트별 서브에이전트 정책을 문서화
