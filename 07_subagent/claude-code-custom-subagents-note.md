# Claude Code 커스텀 서브에이전트(Custom Sub-agents)

## 개요

커스텀 서브에이전트(Custom Sub-agent)는 Claude Code 안에서 특정 역할을 전담하는 독립적인 AI 보조 에이전트이다. 단순히 다른 프롬프트를 쓰는 것과 달리, 서브에이전트는 **별도의 컨텍스트 창(context window)**, **전용 시스템 프롬프트**, **제한된 툴 권한**을 가진 독립 실행 단위이다.

> 비유: 서브에이전트는 팀원이다. 메인 Claude가 PM이라면, 코드 리뷰어·QA 테스터·문서 작성자 등 각 전문가를 필요할 때 불러 쓰는 구조이다.

---

## 핵심 내용

### 1. 내장 서브에이전트 vs 커스텀 서브에이전트

Claude Code에는 기본 내장 서브에이전트가 다섯 가지 있다.

| 에이전트 | `subagent_type` | 역할 | 사용 가능 툴 |
|----------|-----------------|------|-------------|
| **General Purpose** | `general-purpose` (기본값) | 복잡한 멀티스텝 작업, 코드 검색, 리서치 등 범용 | 모든 도구 |
| **Explore** | `Explore` | 코드베이스 탐색 전문. 파일 패턴 검색, 키워드 검색, 구조 파악 | Edit, Write, NotebookEdit, Agent, ExitPlanMode **제외** |
| **Plan** | `Plan` | 구현 전략 설계. 단계별 계획 수립, 아키텍처 트레이드오프 분석 | Edit, Write, NotebookEdit, Agent, ExitPlanMode **제외** |
| **Claude Code Guide** | `claude-code-guide` | Claude Code CLI, Agent SDK, Claude API 관련 질문 답변 | Glob, Grep, Read, WebFetch, WebSearch **만** 사용 |
| **Status Line Setup** | `statusline-setup` | 상태 표시줄 설정 구성 | Read, Edit **만** 사용 |

주요 특징:
- **병렬 실행**: 독립적인 작업은 여러 서브에이전트를 동시에 실행 가능
- **격리 모드**: `isolation: "worktree"`로 별도 git worktree에서 실행 가능
- **재개 가능**: `resume` 파라미터로 이전 에이전트 컨텍스트를 이어받아 후속 작업 수행
- **백그라운드 실행**: `run_in_background: true`로 비동기 실행 가능

커스텀 서브에이전트는 이 다섯 가지 외에 **사용자가 직접 정의**하는 전문화된 에이전트이다. 프로젝트별 또는 전체 환경에서 재사용할 수 있다.

---

### 2. 서브에이전트의 핵심 특징

**독립 컨텍스트 창**
서브에이전트는 부모 대화와 별도의 컨텍스트 창에서 실행된다. 탐색·분석 과정의 중간 결과가 메인 대화에 쌓이지 않으므로, 주 대화의 컨텍스트를 오염시키지 않고 긴 작업을 수행할 수 있다.

**자동 위임(Auto-delegation)**
`description` 필드에 적힌 조건에 맞는 작업이 감지되면 Claude가 자동으로 해당 에이전트에 위임한다. 수동으로 에이전트 이름을 지정할 수도 있다.

**중첩 불가 원칙**
서브에이전트는 다른 서브에이전트를 생성할 수 없다(무한 중첩 방지). 병렬 에이전트 간 통신이 필요하다면 Agent Teams를 사용한다.

### 3. 파일 구조: YAML 프론트매터 + Markdown

서브에이전트는 `.md` 파일로 정의하며, 상단에 YAML 프론트매터를 작성한다.

```markdown
---
name: code-reviewer
description: 코드 품질을 검토하고 개선안을 제안할 때 자동으로 호출된다. PR 리뷰, 리팩토링 제안, 베스트 프랙티스 검토 시 사용한다.
tools: Read, Grep, Glob
model: sonnet
---

당신은 10년 경력의 시니어 개발자이다. 코드를 검토할 때 다음 기준을 따른다:
1. 가독성과 유지보수성을 최우선으로 한다.
2. 각 이슈에 대해 현재 코드 → 문제점 → 개선 코드 순서로 설명한다.
3. 성능 병목 및 보안 취약점을 반드시 확인한다.
```

#### 프론트매터 필드 정리

| 필드 | 필수 여부 | 설명 |
|------|-----------|------|
| `name` | 필수 | 에이전트 식별자 (소문자, 하이픈 허용) |
| `description` | 필수 | **언제 이 에이전트를 호출할지** 명시 — Claude의 자동 위임 판단 근거 |
| `tools` | 선택 | 허용할 툴 목록 (미지정 시 부모 권한 상속) |
| `model` | 선택 | 사용할 모델 (`sonnet`, `haiku`, `opus`) |

### 4. 저장 위치: 범위에 따른 두 가지 경로

```
~/.claude/agents/         ← 사용자 레벨: 모든 프로젝트에서 사용 가능
.claude/agents/           ← 프로젝트 레벨: 현재 프로젝트에서만 사용
```

- **프로젝트 레벨**이 **사용자 레벨**보다 우선한다 (이름 충돌 시).
- 팀 전체가 공유할 에이전트는 `.claude/agents/`에 두고 Git에 커밋한다.
- 개인 워크플로우용 에이전트는 `~/.claude/agents/`에 보관한다.

---

### 5. 생성 방법

#### 방법 ① /agents 명령어 (권장)

```bash
/agents
```

1. `Create new agent` 선택
2. 범위 선택: `User-level` 또는 `Project-level`
3. `Generate with Claude` 선택 후 에이전트 설명 입력
4. `e` 를 눌러 에디터에서 직접 수정 가능

#### 방법 ② 수동 파일 생성

```bash
# 프로젝트 레벨
mkdir -p .claude/agents
touch .claude/agents/my-agent.md

# 사용자 레벨
mkdir -p ~/.claude/agents
touch ~/.claude/agents/my-agent.md
```

### 6. 툴(Tool) 권한 설정

에이전트 역할에 따라 필요한 툴만 부여하는 것이 원칙이다.

| 에이전트 유형 | 권장 툴 | 이유 |
|---------------|---------|------|
| 코드 리뷰어 · 감사자 | `Read, Grep, Glob` | 읽기 전용, 파일 변경 방지 |
| 리서치 · 분석가 | `Read, Grep, Glob, WebFetch, WebSearch` | 정보 수집 필요 |
| 코드 작성자 · 구현자 | `Read, Write, Edit, Bash, Glob, Grep` | 파일 생성 및 실행 필요 |
| 문서 작성자 | `Read, Write, Edit, Glob, Grep, WebFetch, WebSearch` | 리서치 + 문서화 |

툴을 지정하지 않으면 부모 대화의 모든 권한을 상속한다.

---

### 7. 모델 선택 전략

| 모델 | 특징 | 권장 사용처 |
|------|------|------------|
| `haiku` | 빠름 · 저비용 | 단순 탐색, 자주 호출되는 경량 에이전트 |
| `sonnet` | 균형 (기본값) | 대부분의 코드 작성 · 리뷰 에이전트 |
| `opus` | 최고 품질 | 복잡한 아키텍처 설계, 오케스트레이터 |

> 비용 참고: Haiku는 Sonnet 대비 약 3배 저렴하면서 에이전트 코딩 성능의 90%를 유지한다.

### 8. 코드 예제: 실전 커스텀 에이전트 3종

#### ① 코드 리뷰 에이전트 (읽기 전용)

```markdown
---
name: code-reviewer
description: 코드 품질 검토가 필요할 때 호출한다. 가독성, 성능, 보안 관점에서 분석하고 개선안을 제시한다.
tools: Read, Grep, Glob
model: sonnet
---

당신은 시니어 소프트웨어 엔지니어이다.
코드를 검토할 때 아래 순서를 반드시 지킨다:

1. 파일 전체를 읽고 전체적인 구조를 파악한다.
2. 각 이슈마다 [현재 코드] → [문제점] → [개선된 코드] 형식으로 설명한다.
3. 심각도를 🔴 Critical / 🟡 Warning / 🔵 Suggestion 으로 분류한다.
4. 보안 취약점이 발견되면 반드시 최우선으로 보고한다.
```

#### ② 문서 자동화 에이전트

```markdown
---
name: doc-writer
description: README, API 문서, 인라인 주석 작성이 필요할 때 호출한다.
tools: Read, Write, Edit, Glob, Grep, WebFetch
model: sonnet
---

당신은 기술 문서 전문가이다.
- 코드를 읽고 실제 동작을 기반으로 문서를 작성한다. 추측하지 않는다.
- README는 설치 → 사용법 → API 레퍼런스 → 기여 가이드 순서로 작성한다.
- 코드 예시는 반드시 실행 가능한 완전한 형태로 제공한다.
```

#### ③ 테스트 자동화 에이전트

```markdown
---
name: test-writer
description: 테스트 코드 작성, 엣지 케이스 발굴, 테스트 커버리지 분석이 필요할 때 호출한다.
tools: Read, Write, Edit, Bash, Glob, Grep
model: haiku
---

당신은 QA 전문가이다.
- 기존 코드를 분석하여 누락된 엣지 케이스를 찾는다.
- 단위 테스트 → 통합 테스트 순서로 작성한다.
- 외부 의존성은 반드시 mock 처리한다.
- 테스트 실행 후 결과를 확인하고 실패하면 수정한다.
```

### 9. 에이전트 호출 방법

```bash
# 자동 위임 (description 기반으로 Claude가 판단)
> 이 PR의 코드를 리뷰해줘  ← Claude가 code-reviewer를 자동 선택

# 명시적 호출
> code-reviewer 에이전트로 src/auth.py를 분석해줘

# /agents 명령으로 등록된 에이전트 목록 확인
/agents
```

## 자주 하는 실수

| 실수 | 문제 | 해결 |
|------|------|------|
| `description`을 너무 짧게 작성 | Claude가 언제 호출할지 판단 못 함 | "언제, 어떤 상황에서 호출한다"를 구체적으로 명시 |
| 모든 툴을 다 부여 | 에이전트가 의도치 않은 파일 수정 가능 | 역할에 필요한 최소 툴만 부여 |
| 에이전트를 너무 많이 등록 | Claude의 자동 위임 정확도 하락 | 실제로 쓰는 에이전트만 유지 |
| 시스템 프롬프트를 모호하게 작성 | 결과물 품질이 낮고 일관성 없음 | 단계별 지시사항, 출력 형식, 금지 사항을 명확히 기술 |
| 서브에이전트 안에서 다른 에이전트 호출 시도 | 작동하지 않음 (중첩 금지) | Agent Teams 기능을 사용 |

## 핵심 정리

1. **서브에이전트 = 독립 실행 단위**이다. 별도의 컨텍스트 창, 시스템 프롬프트, 툴 권한을 가진다.
2. **`description` 필드**가 자동 위임의 핵심이다. "언제 호출하는지"를 구체적으로 작성해야 Claude가 올바르게 판단한다.
3. **저장 위치**로 범위를 결정한다. 개인용은 `~/.claude/agents/`, 팀 공유용은 `.claude/agents/`에 저장한다.
4. **최소 권한 원칙**을 따른다. 읽기만 필요한 에이전트에 쓰기 권한을 주지 않는다.
5. **모델을 역할에 맞게 선택**한다. 단순 반복 작업은 `haiku`, 복잡한 판단은 `sonnet` 또는 `opus`를 사용한다.

## 관련 개념

- **CLAUDE.md** — 프로젝트 전체 컨텍스트를 Claude에 제공하는 설정 파일
- **Agent Teams** — 여러 에이전트가 병렬로 협업하는 멀티 에이전트 워크플로우
- **Skills** — 에이전트가 참조할 수 있는 재사용 가능한 지식·패턴 문서
- **Slash Commands** — 에이전트 + 스킬을 조합한 워크플로우 자동화 진입점
- **MCP(Model Context Protocol)** — 에이전트의 툴셋을 외부 서비스로 확장하는 프로토콜
