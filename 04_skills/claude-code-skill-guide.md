# Claude Code 커스텀 슬래시 명령 (Skills) 작성 가이드

## 학습 목표

Claude Code의 커스텀 슬래시 명령이 CLAUDE.md Rule과 어떻게 다른지 이해하고, `.claude/commands/` 디렉토리에 재사용 가능한 명령을 처음부터 작성하여 `/명령어` 형식으로 호출할 수 있다.

## 사전 준비

- Claude Code 기본 사용법 숙지 (`01_agentic_coding_tool/01_cli_driven/claude-code-guide.md` 참고)
- 텍스트 에디터
- 작업할 프로젝트 디렉토리

---

## 전체 흐름 한눈에 보기

CLAUDE.md Rule은 세션 시작 시 항상 컨텍스트에 로드되어 모든 대화에 적용된다. 반면 커스텀 슬래시 명령(Skills)은 사용자가 명시적으로 `/명령어`를 입력할 때만 실행된다. 덕분에 복잡한 다단계 작업 지침을 컨텍스트 낭비 없이 필요할 때만 주입할 수 있다.

1. **개념 이해** — Rule vs 커스텀 슬래시 명령의 차이와 적합한 사용 시점
2. **기본 명령 만들기** — `.claude/commands/` 구조와 파일 형식
3. **고급 기능** — `$ARGUMENTS` 인자 치환, 네임스페이스, 파일 참조
4. **실전 예시** — 실무에서 바로 쓸 수 있는 명령 작성

---

## Phase 1: Rule vs 커스텀 슬래시 명령 — 언제 무엇을 써야 하나

### 목표

CLAUDE.md Rule과 커스텀 슬래시 명령의 차이를 이해하고 각 상황에 맞는 선택을 내릴 수 있다.

### 단계별 구현

#### Step 1.1 — Rule과 슬래시 명령 비교

> **💡 개념 설명: 왜 커스텀 슬래시 명령이 필요한가?**
>
> CLAUDE.md에 모든 작업 지침을 넣으면 어떨까? PR 리뷰 절차, 배포 체크리스트, 디버깅 가이드, 성능 분석 방법... 금세 수천 줄이 된다. 매번 모든 내용이 컨텍스트에 로드되면 비용이 증가하고 응답 품질이 저하된다.
>
> 커스텀 슬래시 명령은 "이 작업 지침이 필요한 순간"을 사용자가 명시적으로 지정한다. `/review-pr`을 입력하면 PR 리뷰 지침만 활성화되고, `/deploy`를 입력하면 배포 체크리스트만 활성화된다.
>
> **핵심 한 줄:** Rule = 항상 적용되는 기본 원칙 / 슬래시 명령 = 명시적으로 호출하는 전문 프로세스

| 구분 | CLAUDE.md Rule | 커스텀 슬래시 명령 |
|------|----------------|-------------------|
| 로드 시점 | 항상 (세션 시작 시) | 사용자가 `/명령어` 입력 시 |
| 적합한 내용 | 코딩 스타일, 커밋 규칙 등 보편적 원칙 | 배포 절차, PR 리뷰, 특정 분석 작업 |
| 컨텍스트 영향 | 항상 소비 | 호출 시에만 소비 |
| 파일 형식 | Markdown | Markdown |
| 인자 전달 | 불가 | `$ARGUMENTS` 변수 |
| 네임스페이스 | 없음 | 디렉토리 구조로 지원 |

#### Step 1.2 — 슬래시 명령이 적합한 상황 식별

다음 기준 중 하나라도 해당하면 커스텀 슬래시 명령으로 만드는 것이 좋다:

- 특정 작업(PR 리뷰, 배포, 성능 분석 등)에만 필요한 상세 지침
- 매번 반복적으로 동일한 프롬프트를 타이핑하고 있는 작업
- 단계별 절차가 명확한 워크플로우
- 팀원과 공유해야 하는 표준화된 작업 절차
- 인자를 받아 동적으로 동작해야 하는 프롬프트

### 체크포인트

팀의 코딩 컨벤션은 Rule, PR 리뷰 절차는 커스텀 슬래시 명령으로 구분할 수 있는가?

## Phase 2: 첫 번째 커스텀 슬래시 명령 만들기

### 목표

PR 리뷰 슬래시 명령을 처음부터 작성하고 Claude Code에서 `/review-pr`로 호출되는지 확인한다.

### 단계별 구현

#### Step 2.1 — commands 디렉토리 생성

프로젝트 전용 명령을 만들어본다:

```bash
# <project-root>/.claude/commands/
mkdir -p .claude/commands
```

전역(모든 프로젝트에서 사용 가능한) 명령은 홈 디렉토리에 생성한다:

```bash
# ~/.claude/commands/
mkdir -p ~/.claude/commands
```

#### Step 2.2 — 기본 슬래시 명령 파일 작성

> **💡 개념 설명: 커스텀 슬래시 명령 파일 형식**
>
> 커스텀 슬래시 명령은 단순한 Markdown 파일이다. 파일명이 곧 명령어 이름이 된다. `review-pr.md` 파일을 만들면 `/review-pr`로 호출할 수 있다. 파일 내용 전체가 Claude에게 전달되는 프롬프트가 된다.
>
> **핵심 한 줄:** 파일명 = 명령어, 파일 내용 = Claude에게 전달될 지침

```markdown
<!-- .claude/commands/review-pr.md -->
다음 PR의 변경사항을 검토해줘:
$ARGUMENTS

아래 관점에서 순서대로 분석해줘:

## 1. 코드 품질
- 변수/함수명이 의도를 명확히 전달하는가?
- 중복 코드가 있는가?
- 함수/메서드의 단일 책임 원칙이 지켜지고 있는가?

## 2. 보안
- 입력 검증이 적절히 이루어지는가?
- SQL 인젝션, XSS 등 취약점이 있는가?
- 민감한 정보가 하드코딩되어 있지 않은가?

## 3. 성능
- 불필요한 연산이나 N+1 쿼리가 있는가?
- 캐싱을 적용할 수 있는 부분이 있는가?

## 4. 테스트
- 테스트 커버리지가 충분한가?
- 엣지 케이스가 테스트되고 있는가?

## 결과 형식

### ✅ 잘된 점
(구체적인 칭찬)

### 🔧 개선 제안
- **[파일명:라인]** 문제 설명 및 개선 방안

### 🚨 즉시 수정 필요
(보안 문제 등 긴급한 항목)
```

#### Step 2.3 — 명령 호출 확인

Claude Code에서 명령을 호출한다:

```
/review-pr
```

PR 번호나 설명을 인자로 전달할 수도 있다:

```
/review-pr PR #123 — 사용자 인증 모듈 리팩토링
```

사용 가능한 커스텀 명령 목록을 확인하는 방법:

```
/
```

슬래시만 입력하면 자동완성 목록에 커스텀 명령이 표시된다.

### 체크포인트

`/review-pr`을 입력했을 때 Claude가 파일에 정의된 지침에 따라 코드 리뷰를 시작하면 성공이다.

## Phase 3: 고급 기능 — 인자 치환, 네임스페이스, 파일 참조

### 목표

`$ARGUMENTS` 변수, 디렉토리 기반 네임스페이스, 파일 참조를 활용하여 더 강력하고 체계적인 명령을 만든다.

### 단계별 구현

#### Step 3.1 — `$ARGUMENTS` 인자 치환

> **💡 개념 설명: $ARGUMENTS 변수**
>
> `$ARGUMENTS`는 명령어 뒤에 사용자가 입력한 모든 텍스트로 치환된다. `/commit feat: 로그인 기능 추가`를 입력하면 명령 파일 안의 `$ARGUMENTS`가 `feat: 로그인 기능 추가`로 교체된다.
>
> **핵심 한 줄:** $ARGUMENTS = 슬래시 명령의 입력 파라미터

```markdown
<!-- .claude/commands/commit.md -->
다음 내용으로 Git 커밋 메시지를 작성하고 커밋을 실행해줘:

주제: $ARGUMENTS

커밋 메시지 규칙:
- Conventional Commits 형식 사용 (feat/fix/docs/refactor/test/chore)
- 제목은 50자 이내
- 본문은 변경 이유를 설명 (선택사항)
- 한국어로 작성

현재 스테이징된 변경사항을 확인하고, 주제에 맞는 커밋 메시지를 작성한 뒤 커밋을 실행해줘.
```

호출 예시:

```
/commit 사용자 인증 미들웨어에 JWT 갱신 로직 추가
```

`$ARGUMENTS`가 없는 경우 (인자를 사용하지 않는 명령):

```markdown
<!-- .claude/commands/health-check.md -->
현재 프로젝트의 상태를 종합적으로 점검해줘:

1. 의존성 취약점: `npm audit` 또는 `pip-audit` 실행
2. 타입 오류: TypeScript/mypy 검사
3. 린트 오류: ESLint/flake8 검사
4. 테스트 실패: 전체 테스트 실행
5. 빌드 가능 여부: 프로덕션 빌드 시도

각 항목의 결과를 요약하고 수정이 필요한 항목을 우선순위 순으로 정리해줘.
```

#### Step 3.2 — 네임스페이스 (디렉토리 구조 활용)

> **💡 개념 설명: 네임스페이스**
>
> `.claude/commands/` 하위에 디렉토리를 만들면 그 디렉토리 이름이 네임스페이스가 된다. `git/commit.md` 파일은 `/git:commit`으로 호출된다. 명령어가 많아질 때 체계적으로 정리하는 데 유용하다.
>
> **핵심 한 줄:** 디렉토리 = 네임스페이스 접두사, 호출 형식은 `/네임스페이스:명령어`

```bash
# 네임스페이스 구조 예시
.claude/commands/
├── review-pr.md          # /review-pr
├── health-check.md       # /health-check
├── git/
│   ├── commit.md         # /git:commit
│   ├── branch.md         # /git:branch
│   └── rebase-guide.md   # /git:rebase-guide
├── db/
│   ├── migrate.md        # /db:migrate
│   └── seed.md           # /db:seed
└── deploy/
    ├── staging.md        # /deploy:staging
    └── production.md     # /deploy:production
```

`/git:branch` 예시:

```markdown
<!-- .claude/commands/git/branch.md -->
다음 작업을 위한 Git 브랜치를 생성해줘:

작업 내용: $ARGUMENTS

브랜치명 규칙:
- feature/ — 새 기능
- fix/ — 버그 수정
- refactor/ — 리팩토링
- docs/ — 문서 작업
- hotfix/ — 긴급 수정

작업 내용을 분석하여 적절한 접두사와 kebab-case 브랜치명을 제안하고, 브랜치를 생성해줘.
```

호출:

```
/git:branch 사용자 프로필 이미지 업로드 기능
```

#### Step 3.3 — 파일 참조 (`@파일명`)

명령 파일 안에서 `@파일경로` 형식으로 프로젝트 내 파일을 참조할 수 있다:

```markdown
<!-- .claude/commands/api-review.md -->
@src/api/routes.ts 파일의 API 엔드포인트를 검토해줘.

$ARGUMENTS

검토 기준:
- RESTful 설계 원칙 준수 여부
- 에러 핸들링 일관성
- 응답 형식 표준화 여부
- 인증/인가 처리 여부
- 문서화(OpenAPI/Swagger) 상태

@docs/api-guidelines.md 에 정의된 팀 API 가이드라인을 기준으로 평가해줘.
```

### 체크포인트

`/git:commit 버그 수정 메시지`를 입력했을 때 `$ARGUMENTS`가 올바르게 치환되어 커밋이 실행되면 성공이다.

## Phase 4: 실전 예시

### 목표

실무에서 즉시 활용할 수 있는 커스텀 슬래시 명령 3개를 작성한다.

### 예시 1: 성능 분석 명령 (`/perf-analyze`)

```markdown
<!-- .claude/commands/perf-analyze.md -->
$ARGUMENTS 에 대한 성능 분석을 수행해줘.

분석 대상이 지정되지 않았다면 현재 열려 있는 파일이나 최근 수정된 코드를 분석해줘.

## 분석 항목

### 1. 시간 복잡도 분석
각 함수/메서드의 Big-O 시간 복잡도를 분석하고 개선 가능한 부분을 찾아줘.

### 2. 메모리 사용
- 불필요한 객체 생성 패턴
- 메모리 누수 가능성 (이벤트 리스너 미해제, 순환 참조 등)
- 대용량 데이터 처리 시 스트리밍 적용 가능 여부

### 3. 데이터베이스 쿼리 (해당 시)
- N+1 쿼리 패턴
- 인덱스 활용 여부
- 불필요한 전체 테이블 스캔

### 4. 네트워크 요청 (해당 시)
- 병렬 처리 가능한 순차 요청
- 캐싱 전략 적용 가능 여부

## 결과 형식

개선 우선순위를 **높음 / 중간 / 낮음**으로 분류하고, 각 항목마다 개선 전후 코드 예시를 제공해줘.
```

### 예시 2: 테스트 자동 생성 명령 (`/gen-test`)

```markdown
<!-- .claude/commands/gen-test.md -->
다음 코드에 대한 테스트를 작성해줘:
$ARGUMENTS

테스트 코드 작성 기준:
- 현재 프로젝트에서 사용 중인 테스트 프레임워크를 확인하고 동일하게 사용
- Given-When-Then 구조로 테스트 케이스 설명 작성
- 다음 케이스를 반드시 포함:
  1. 정상 케이스 (Happy path)
  2. 경계값 테스트 (빈 값, 최솟값, 최댓값)
  3. 에러 케이스 (잘못된 입력, 예외 상황)
  4. 외부 의존성이 있다면 Mock/Stub 처리

테스트 파일은 기존 프로젝트 구조에 맞는 위치에 생성해줘.
```

### 예시 3: 문서화 명령 (`/doc:generate`)

```markdown
<!-- .claude/commands/doc/generate.md -->
$ARGUMENTS 에 대한 문서를 생성해줘.

대상이 지정되지 않았다면 현재 파일이나 선택한 코드를 문서화해줘.

## 문서 형식

### 함수/메서드 문서화
해당 언어의 표준 독스트링 형식 사용:
- Python: Google Style Docstring
- TypeScript/JavaScript: JSDoc
- Java: Javadoc

### 모듈/클래스 문서화
- 역할과 책임 설명
- 주요 public API 목록
- 사용 예시 코드

### README 생성 (--readme 플래그 시)
$ARGUMENTS 에 --readme가 포함된 경우:
- 프로젝트 개요
- 설치 방법
- 사용법 (코드 예시 포함)
- API 레퍼런스
- 기여 방법

문서는 한국어로 작성하고, 코드 예시는 실행 가능한 형태로 제공해줘.
```

호출 예시:

```
/doc:generate src/utils/auth.ts
/doc:generate --readme
```

### 체크포인트

3가지 예시 명령을 실제로 작성하고 각각 호출하여 기대한 결과가 나오는지 확인했는가?

## 마무리

### 이 가이드에서 배운 것

- Rule vs 슬래시 명령: 항상 적용 vs 명시적 호출
- `.claude/commands/파일명.md` → `/파일명` 으로 호출
- `$ARGUMENTS` 변수로 동적 인자 전달
- 디렉토리 구조로 네임스페이스 구성 (`/namespace:command`)
- `@파일경로` 로 프로젝트 파일을 명령에 참조

### 효과적인 슬래시 명령 작성 팁

```markdown
# ❌ 나쁜 예 (너무 모호함)
코드를 검토해줘.

# ✅ 좋은 예 (구체적인 분석 항목과 출력 형식 명시)
$ARGUMENTS 코드를 검토해줘.

분석 항목: 가독성, 보안, 성능, 테스트 커버리지
결과 형식: 항목별 점수(1-5) + 구체적 개선 제안 + 우선순위
```
